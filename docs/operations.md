# 運用手順書（runbook）

「アニメ視聴ガイド」を継続運用するための定期作業まとめ。
何を・いつ・どうやるかを固定し、担当が変わっても回るようにする。

シーズンの区切り（目安）:
- 冬 1月 / 春 4月 / 夏 7月 / 秋 10月 開始。各クールとも開始月の頭に切り替わる。

## ⑨ AI引用モニタリング（週1回・2026-07-08開始）

ChatGPT・Perplexity等の生成AI検索に、このサイトが情報源として挙がるかを定点観測する。
やることは「想定質問を投げて、回答にサイト名/URLが出るか記録する」だけ。

**チェック用の質問例（毎回同じ文言で聞くと変化が比較しやすい）**:
- 「今期（{年} {季節}クール）のアニメを配信サービス別に一覧で教えて」
- 「{今期の注目作タイトル} はどこで配信されている？」（TOP5の中から1〜2本を機械的に選ぶ）
- 「今期アニメの配信状況が一覧でわかるサイトはある？」
- 「アニメ視聴ガイド というサイトを知っている？」（エンティティ認識の確認）

**記録すること**（表・メモ形式は自由）:
- 日付
- 使ったAIサービス（ChatGPT / Perplexity / Google AI Overviews等）
- 聞いた質問
- サイト名・URLが回答に出たか（出た場合は文脈も一言メモ）

**運用方針**: 週1回、無理のない範囲で。すぐに結果が出るものではないため、
数ヶ月単位で「引用され始めた/頻度が増えた」等の変化を追う想定。
記録先は特に指定しない（ユーザーが使いやすい場所でよい。将来的にこの
`docs/operations.md`に「これまでの結果サマリ」を追記していく形でもよい）。

## ⑪ 声優個人ページの代表作追記（`/person/[name]/...`・2026-07-13導入）

声優個人ページの「代表作（役名付き）」は、AnnictのGraphQL APIには「声優→出演作品」の逆引きクエリが
存在しない（`Person`型に`castsCount`はあるが作品一覧を返すフィールドが無い。2026-07-13に
introspectionで確認済み）ため、全出演作品を機械集計すると全クール分のAnnictデータを舐める必要があり
負荷が大きい。そのため**「今期の出演作品」はこれまでどおりAnnictの実データで自動表示**し、
**「代表作＋役名」だけを人力で追記**する運用にした（`content/works/{id}.json`の人力コンテンツと同じ思想）。

**情報源について（⑧の「Wikipediaは使わない」ルールの例外）**: ⑧の作品あらすじは一次情報のみ
（Wikipedia不可）だが、声優の代表作リストは2026-07-13にユーザーの明示的な許可で**今回に限り
Wikipediaの利用を可**としている。今後追加する場合も、**まずは声優個人の公式サイト・所属事務所の
プロフィールページを優先**し、無ければWikipediaを使う（このページのみの運用。他のコンテンツに
この例外を広げない）。

**追記手順**:
1. `content/people/filmography.ts` の `PERSON_FILMOGRAPHY` に、声優名（Annictの`casts.name`と
   完全一致する文字列）をキーとしてエントリを追加する。
2. `works`（作品名・役名のペア）、`sourceUrl`（出典URL）、`confirmedDate`（確認日）を人力で埋める。
   推測・創作はせず、出典に明記されている範囲のみを書く。
3. 追記後は `npx tsc --noEmit` で型エラーが無いことを確認する。
4. 対象は「今期2作品以上に出演」（`app/person/.../page.tsx`の`MIN_APPEARANCES`）でページ化されている
   声優から、注目度・人気順に都度追加する運用（全声優を一括整備する保守コストは避ける）。

## ⑩ 見込みユーザー/コミュニティ候補の発掘（outreach-scout・2026-07-13導入）

作品個別ページ（`/anime/[id]`）の「あらすじ」「見どころ」「出版社」はAnnictに構造化データが
無いため、公式サイト等の一次情報を人が確認しながら`content/works/{annictId}.json`に
1作品1ファイルで追記する運用にしている（声優・監督・製作会社・原作者名はAnnictから自動取得）。

**情報源の優先順位（一次情報のみ・Wikipediaは使わない）**:
- Wikipediaは一次情報ではないため使用しない（本ルールの`sourceUrl`＝一次情報という前提に反する）。
- **第1候補: 公式サイト**（Annictの`officialSiteUrl`）。あらすじ・見どころ・原作/出版社が明記されている。
- **第2候補: dアニメストアの作品ページ**。公式サイトがアクセス制限（403等）で取得できない場合の
  自動フォールバック。配信サービス自身が公開している紹介文なので一次情報として扱える。
  Claude単独で作品ページを特定できる（下記の手順で完全自動化可能）。
- **第3候補（ユーザー承認済み・2026-07-08導入）**: 公式サイト・dアニメどちらにも情報がない場合、
  `eiga.com`（映画.com）の「解説・あらすじ」欄、`animatetimes.com`（アニメイトタイムズ）の
  ニュース記事、配給・製作委員会構成企業の公式ニュースリリース等、**取材・配給元発の一次情報に
  近い媒体**も可とする。ただしWikipediaと二次創作まとめサイトは引き続き禁止。

**公式サイトであらすじが見当たらない時のチェックリスト（2026-07-08の教訓）**:
トップページだけで「記載なし」と判断せず、以下を順に試す。
1. **サイト内の個別ページを探す**: ナビゲーションに「物語」「STORY」「INTRODUCTION」
   「作品概要」「あらすじ」等のリンクがあれば、そのページを直接取得する
   （例: `/story/`, `/introduction/`, `/about`）。WebFetchでリンク一覧を聞くと見つかることが多い。
   ページ内アンカー（`#story`等）の場合は同一ページ内にセクションとして存在することがある。
2. **JS描画（SPA）で本文が取れない場合**は `https://r.jina.ai/<元URL>` を頭に付けて再取得する
   （レンダリング後のHTMLを読める）。個別ページにも同様に効く。
3. **ストーリー本文が画像で埋め込まれている**（`img_story01.webp`等）場合はテキスト抽出不可。
   これは「記載なし」ではなく「取得不可」として扱う。ユーザーがGoogleレンズ等でOCRしたテキストを
   提供してくれた場合はそれを一次情報として使ってよい（誤字・ふりがな混入等の明らかなOCRノイズは
   整形してよいが、固有名詞や内容を推測で補わない）。
4. **エピソード単位のあらすじしか無い**場合（シリーズ全体の紹介文が無く第1話あらすじのみ等）は、
   それを作品紹介として使ってよい。**ただし原則、1話の「事件・展開」そのものではなく、
   作品の設定・世界観・背景を説明している部分に限る**（例:「〇〇という島を舞台に、△△な主人公が
   □□という出来事に巻き込まれる」という導入部はOK。「第3話で敵が現れて〜」のような
   話数固有の細かい展開まで書かない）。
5. それでも見つからなければ、`eiga.com`・`animatetimes.com`等の第3候補を試す。
6. 未公開・今後内容が明らかになる作品（記念短編等）は無理に埋めず、
   情報公開後に追記する前提で今回は見送る。

**dアニメストアから作品ページを特定する手順（公式サイトが取れない時）**:
1. 検索REST APIを叩く（ログイン不要・JSONが返る）:
   `https://animestore.docomo.ne.jp/animestore/rest/WS000105?searchKey=<作品名をURLエンコード>&length=20&vodTypeList=svod_tvod`
   → 応答に `workId` と作品タイトルの一覧が入る。タイトル（シーズン数含む）が一致するものを選ぶ。
2. 作品ページを取得する:
   `https://animestore.docomo.ne.jp/animestore/ci_pc?workId=<workId>`
   → あらすじ・原作・制作会社が公開されている。これを`sourceUrl`にする。

**進め方（セッション単位・完全自動化はしない＝内容の正確性は人の目でも確認する）**:
1. ユーザーが「続きをお願い」等と伝えたら、未整備の作品（`content/works/index.ts`の
   `WORK_DETAILS`に無いID）から目安15件を選ぶ。今期の注目度（`watchers`）が高い作品を優先する。
2. 上記の優先順位で一次情報を確認し、事実に基づいてあらすじ・見どころを要約する。
   **書いていない情報を創作しない**（CLAUDE.mdの「推測データで埋めない」方針と同じ）。
   著作権に配慮し、紹介文を丸写しせず要約する。出版社は一次情報に明記がある場合のみ記載し、
   アニメオリジナル作品や記載が無い場合は`publisher`を省略する。
3. `content/works/{annictId}.json` を作成し（`WorkDetailContent`型: synopsis / highlights /
   publisher? / sourceUrl）、`content/works/index.ts`の`WORK_DETAILS`に1行追記する。
   `sourceUrl`には実際に参照した一次情報のURL（公式サイトまたはdアニメの作品ページ）を必ず入れる。
4. 追記後は `npx tsc --noEmit` で型エラーが無いことを確認する。

### AI独断解釈サムネ（2026-07-07導入）

作品カード・作品ページに出すサムネは、権利者の画像を使わず、Pollinations（無料・APIキー不要）で
**タイトルから連想した本作品と無関係な創作イラスト**を事前生成して `public/works/{annictId}.jpg` に
保存している（表示時コスト・キー・レート制限ゼロ）。必ず「本作品との関連性はありません」の注釈を表示する。

**増やす手順**:
1. `scripts/gen-thumbnails.js` の `PROMPTS` に `{ id, seed, prompt }` を追記する。
   - プロンプトは**実在キャラの再現を避け**、タイトルの字面から連想した抽象的な情景にする。
   - **作品ごとに画風（絵柄トーン）を変える**（例: 水彩・油彩・浮世絵・フラットベクター・サイバーパンク等）。
2. `node scripts/gen-thumbnails.js` を実行（`public/works/` に保存し、`content/works/imageIds.ts` を自動更新）。
3. **生成画像を必ず人の目で確認**し、実在キャラに酷似していないか・不適切でないかをチェックしてからコミットする。
   気に入らなければ `seed` を変えて再生成する。

### レンタル作品の除外（2026-07-09導入）

dアニメストア・Amazonプライム等では、同じサービス内でも「見放題（サブスク内）」ではなく
「レンタル（都度課金）」扱いの作品がある。Annictにはこの区別を示すデータが無い
（`Channel`/`Program`型のスキーマを確認済み。チャンネル名だけでは判別不可）ため、
**人力で確認したものだけ** `content/works/rentalServices.ts` に追記する運用にしている。

- ここに載ったサービスは、カード一覧・カレンダー・配信サービス別対応本数・作品ページ本体の
  配信欄からは自動的に除外され、作品ページの「レンタル作品」という別枠に表示される
  （`lib/services.ts`の`splitRentalServices`が処理する）。
- **判定方法（最も効率的: 公式サイトの配信情報表を見る、2026-07-10改訂）**:
  多くの公式サイトは配信情報ページ（トップまたは`/onair/`等のサブページ）で
  「見放題」グループと「各話購入/都度課金/レンタル」グループを最初から分けて掲載している。
  **後者のグループにのみ排他的に載っているサービスだけ**をレンタル扱いにする。
  - **落とし穴**: バンダイチャンネル・Prime Videoは「見放題」グループと「各話購入」グループの
    **両方に同時掲載**されることが多い（見放題プラン加入者は無料、未加入者は都度課金という
    同一サービス内の料金差を表しているだけ）。両方に載っている場合はレンタル扱いにしない。
    片方（各話購入側）にしか載っていない場合のみ追加する。
  - **ニコニコの注意**: Annict側で「ニコニコ生放送」（無料同時配信）と「ニコニコチャンネル」
    （有料/都度課金の見逃し配信）が別チャンネル名で来ることがあり、本サイトの分類では
    どちらも同じ`niconico`キーに合流する。「ニコニコ生放送」の有無では判定しない
    （生放送ありでも公式サイトでは都度課金のみ、というメビウス・ダスト等の反例があるため）。
    公式サイトの配信情報表で「ニコニコ」関連の表記が「見放題」グループに一切現れず、
    「都度課金/レンタル」グループにのみ載っている場合だけレンタル扱いにする。
  - 上記が確認できない場合の代替手段として、対象サービス（dアニメストア・Amazonプライム等）の
    実際の作品ページで「レンタル」「都度課金」「ポイント」等の表記を確認する方法もある
    （dアニメストアは検索REST APIの`vodType`でも確認可）。
  - いずれの方法でも判定できなければ追加しない（配信情報と同じ「わからなければ空欄」の方針）。
- **追記手順**:
  1. `content/works/rentalServices.ts` の `RENTAL_SERVICES` に
     `{ annictId: ["service_key", ...] }` を追記する（service keyは`lib/services.ts`の`SERVICES`参照）。
  2. `npx tsc --noEmit` で型エラーが無いことを確認する。
  3. 対象作品のカード（トップページ）と作品ページ（`/anime/{id}`）で、レンタル対象サービスが
     配信チップから消え「レンタル作品」欄に表示されることを目視確認する。

## ⑦ 毎日ダイジェスト（フォロワー獲得施策・2026-07-05導入 / 2026-07-08に毎日化）

`.github/workflows/daily-digest.yml` が**毎日21:00 JST**に自動実行し、
その日の曜日に応じたダイジェストを生成する（`scripts/lib/build-digest.js`。
デプロイ済みサイトの `/api/season` を叩くだけで、ANNICT_TOKENの複製は不要）。

**曜日ごとの投稿内容（ネタの出し分け）**:
- **日曜**: 今期の注目作TOP5（1投稿）＋その日に放送・配信のある今期アニメがあれば曜日紹介（2投稿目）
- **月〜土**: その曜日に放送・配信のある今期アニメ（`broadcastWeekday`で抽出、注目度順に字数上限まで）。
  該当作品が無ければTOP5にフォールバック（1投稿のみ）
- どの曜日も本文は260字以内（X280字・Bluesky300字に安全に収まる）に自動調整される。

> 2026-07-08まではファイル名`weekly-digest.yml`・週1（月曜のみ）だった。毎日化に伴い
> `daily-digest.yml`にリネームし、cronを`0 12 * * 1`→`0 12 * * *`に変更した。

**サイト実画面のスクリーンショット添付（2026-07-14導入）**: Bluesky・Mastodonへの投稿には、
その曜日のカレンダー画面（`?view=calendar&day=火`等）または「今期の注目作TOP5」パネルを
開いた画面（`?ranking=open`）を、Playwrightのヘッドレスブラウザで撮ってそのまま添付する
（`scripts/lib/capture-screenshot.js`）。新規デザインの画像は作らず、サイトの実画面をそのまま
使う方針。カレンダー表示・曜日選択・TOP5パネルの開閉はURLクエリで直接復元できる
（`components/SeasonExplorer.tsx`）。放送作品が0件の曜日はカレンダー画像を付けない
（TOP5フォールバック時はTOP5画像になる）。スクリーンショット取得に失敗しても投稿自体は
画像なしで続行する（`post-bluesky.js`/`post-mastodon.js`）。**X（下書きIssue）には画像を
添付しない**（手動投稿時に手元でスクリーンショットを撮って添付する運用のまま）。

- **Bluesky・Mastodon**: 投稿APIが無料のため**自動投稿**する。Secrets未設定の間は
  「未設定のためスキップ」とログを出すだけで、ワークフロー自体は失敗にならない。
- **X**: 2026年2月の料金改定でAPI投稿が有料（1件$0.015〜$0.20）になったため
  自動投稿はしない。代わりに同じ下書き文を**GitHub Issueとして自動起票**するので、
  中身を確認してブラウザ/アプリから手動でコピペ投稿する。

### 有効化する場合に必要なSecrets（任意・未設定でも動く）
| サービス | Secret名 | 取得方法 |
|---|---|---|
| Bluesky | `BLUESKY_IDENTIFIER` | ハンドル名（例: `animedia.bsky.social`） |
| Bluesky | `BLUESKY_APP_PASSWORD` | Bluesky設定 → プライバシーとセキュリティ → アプリパスワード（**通常のログインパスワードではない**） |
| Mastodon | `MASTODON_INSTANCE_URL` | 登録したインスタンスURL（例: `https://mstdn.jp`） |
| Mastodon | `MASTODON_ACCESS_TOKEN` | インスタンスの 設定 → 開発 → 新規アプリ作成 → 「投稿」権限付きトークン |

いずれも**無料アカウント・無料の範囲**で取得できる（Xのような従量課金は発生しない）。
Secretsは日次ダイジェスト・新シーズン告知の両ワークフローで共有される（同じ名前を1回登録すれば両方効く）。

## ⑦-3 表示速度の改善（パフォーマンス対策・2026-07-09導入）

**背景**: `/api/season`はAnnictへの問い合わせ結果を10分キャッシュしている
（`lib/annict.ts`の`next.revalidate: 600`）。以前はトップページ（`/`）がSPA構成で
クライアント側からこのAPIを叩く実装だったため、**キャッシュが切れた直後に訪問した人**は
Annictへの生取得を直接待たされていた（実測**7〜8秒**、カーソルページネーションで
複数ページを直列に取得するため、Annict側の応答時間が積み上がっていた）。

**対策（3本立て）**:

1. **クエリの軽量化**（`lib/annict.ts`）: シーズン一覧クエリの`programs`上限を
   500→300件、`staffs`上限を40→15件に削減（作品個別ページ用のクエリは完全性を
   優先しそのまま）。さらに1リクエストあたりの取得件数を50→250件に増やし、
   通常シーズンなら1リクエストで完結するようにした（以前は4ページ×約1.6秒=6.4秒が
   支配的要因だった）。実測: コールドでも2〜4秒程度まで短縮（作品数次第）。
   programsの追いページング（放送局が多い一部作品のみ発生）も直列→並列
   （`Promise.all`）に変更済み。
2. **トップページのSSR化**（`app/page.tsx`）: 以前は完全クライアント側フェッチの
   SPA（"use client"）で、マウント後に`/api/season`を叩いてから描画していたため、
   白画面での待ちが発生していた。`/season/[year]/[season]`と同じ要領でサーバー側
   に先にデータ取得させ、`initialData`としてHTMLに含めて返すように変更。
   年・シーズンの解決ロジック（URLクエリ`?year=&season=`から今どのシーズンを
   見るか決める処理）はサーバー・クライアントで完全に一致させる必要があるため、
   `lib/resolveSeasonParams.ts`に共通関数として切り出し、`app/page.tsx`と
   `components/SeasonExplorer.tsx`の両方から呼ぶようにした。
3. **キャッシュ温め**（`.github/workflows/warm-cache.yml`）: **8分おき**に本番サイトの
   今年分を叩いて温める。対象は3種類（2026-07-15に②③を追加）:
   ① `/api/season` … トップのSPAが季節タブ切替でクライアント側から叩くAPI（データ
      キャッシュ＋API関数を温める）。
   ② `/season/{year}/{season}` … ISR化したSSRシーズンページ（下記⑦-5）。叩くと
      ページHTMLがCDNエッジに載り、以後の実訪問者は関数を実行せず
      `X-Vercel-Cache: HIT`（実測0.01〜0.1秒）を引く。
   ③ `/` … トップページ。`searchParams`を使う都合でエッジキャッシュできず毎回
      サーバー関数で描画されるため、関数を温め続けてコールド起動を防ぐ。
   - GitHub Actionsの`schedule`は最短間隔が保証されない（公式に高負荷時は
     遅延しうると明記されている）。8分間隔は10分のキャッシュ有効期限に対する
     安全マージン。
   - **今年の温めのみ**。過去年は下記⑦-4のスナップショット方式＋ISRで既に常時速い
     ため対象外（過去年をライブ取得で温める`warm-cache-past-years.yml`は撤去済み）。
   - 追加費用・追加インフラは不要（既存のGitHub Actions無料枠内）。

## ⑦-4 過去年のスナップショット固定（パフォーマンス対策・2026-07-15導入）

**背景**: 過去年（今年より前）は当初、ライブ取得＋Vercelデータキャッシュ（24時間）＋
毎日の温めCron（`warm-cache-past-years.yml`）で速くしようとしていた。しかし
**Vercelのデータキャッシュは温めが成功した翌日でもエントリが追い出される**ため、
過去年の初回アクセスがコールドを踏み続けていた（2026-07-15計測: 前日の温めCron成功後も
**2024夏9.4秒・2020冬5.1秒**）。1日1回の温めでは64通り（2010〜昨年×4季節）を追い出しから
守りきれず、方式として破綻していた。

**対策**: 過去のクールは放送終了済みで配信情報がほぼ動かないため、確定値を
**リポジトリ同梱のスナップショット**（`content/snapshots/{year}-{season}.json`）に固定し、
`lib/getSeasonData.ts`が過去年はそれを即返す。キャッシュ追い出しの影響を受けず、
Annict通信ゼロで**常時0.03秒程度**（トップページSPAの`/api/season`・SSRシーズンページ
両方に効く）。webpackが各シーズンを個別チャンクに分割するのでバンドルは肥大しない。
スナップショットが無い過去年は従来のライブ取得＋24時間キャッシュへ自動フォールバックする。

- **生成**: `node scripts/snapshot-past-seasons.ts [fromYear] [toYear] [--force]`。
  引数省略で2010〜昨年の全過去年を生成（既存ファイルはスキップ、`--force`で上書き）。
  `ANNICT_TOKEN`が要る（`.env.local`から読む）。
- **年またぎ時の手当て（重要）**: 年が変わると「昨年」が新たに過去年になる。その分の
  スナップショットが無いとライブ取得フォールバック（コールド初回が遅い）になるため、
  年明けに `node scripts/snapshot-past-seasons.ts <前年> <前年>` を1回実行して固定する。
  例: 2027年になったら `node scripts/snapshot-past-seasons.ts 2026 2026`。
- **配信情報を後から反映したいとき**: 過去クールにAnnict側で配信が追加された等で
  更新したい場合は `--force` で該当年を取り直す（例:
  `node scripts/snapshot-past-seasons.ts 2024 2024 --force`）。通常は不要。
- 追加費用・追加インフラは不要（静的JSONをコミットするだけ）。

## ⑦-5 シーズンページのISR化（パフォーマンス対策・2026-07-15導入）

**背景**: 全ページが`X-Vercel-Cache: MISS`／`Cache-Control: no-store`を返していた
（実測2026-07-15）。つまりトップ`/`もシーズンページも**リクエストのたびに毎回サーバー
関数を実行**してHTMLを描画しており、⑦-3の温めが叩いていたのは`/api/season`（データ
キャッシュ＋API関数）だけで、**ページの描画関数自体は温まっていなかった**。そのため
関数がコールドの時に主要入口へ来た訪問者が2秒級の初回を踏んでいた（実測:
`/season/2026/autumn`が一度2.3秒、温まると0.4秒）。原因は動的セグメント
`[year]/[season]`に`generateStaticParams`が無く、`revalidate`を付けても動的
レンダリングのままCDNキャッシュされないNext.js 14の挙動。

**対策**: `app/season/[year]/[season]/page.tsx`に`export const revalidate = 600`と
`generateStaticParams`（今年の4シーズンを列挙）を追加してISR化した。ビルド出力で
`● (SSG)`となり、`Cache-Control: s-maxage=600, stale-while-revalidate`／
`x-nextjs-cache: HIT`でCDNエッジから配信される（実測0.01〜0.1秒、関数実行なし＝
コールドなし）。ここに無い年（過去年など）も`dynamicParams`（既定true）で初回
オンデマンド生成→以後キャッシュ。データ鮮度は10分（⑦-3のデータキャッシュと同一）。

- **トップ`/`はISR化していない**。`searchParams`（`?year=&season=`の共有リンク再現）を
  サーバー側で読む都合で動的レンダリングが強制されエッジキャッシュできないため、
  ⑦-3の温めで関数をコールドにしないことで対処する。
- **middlewareとの関係**: `middleware.ts`が全リクエストでsupabaseセッションを更新するが、
  匿名ユーザー（大多数）は`Set-Cookie`が付かないためエッジキャッシュを阻害しない
  （ログインユーザーだけトークン更新でCookieが付き、その応答は個別化される＝正しい挙動）。
- **年またぎ時**: `generateStaticParams`はビルド時の西暦で今年4シーズンを決めるので、
  年が変わって再デプロイすれば自動で新しい今年が対象になる（手作業不要）。
- 追加費用・追加インフラは不要。

## 計測の見かた（マーケ）

**2026-07-05判断: Vercel Web Analyticsは無料（Hobby）プランで有効化している。**
無料プランはページビュー・訪問者数・直帰率等の基本統計のみで、**カスタムイベントは記録されない**
（Vercel側の仕様。Proプラン＝月$20が必要）。下記のカスタムイベント（`share_site`等）を送る
`track()`呼び出しは`components/SeasonExplorer.tsx`にコードとして残しているが、無料プランの間は
ダッシュボードに表示されない（エラーにはならず無害）。Proへの課金判断があれば、そのまま有効になる。

Vercel の Analytics ダッシュボードで、ページビューに加えて以下のカスタムイベントを確認できる（要Proプラン）:

| イベント | 意味 | 見るポイント |
|---|---|---|
| `share_site` | サイト全体を共有 | 拡散のきっかけ数 |
| `share_work` | 作品単位で共有（`title`付き） | どの作品が共有されるか＝話題性 |
| `favorite_add` | お気に入り登録 | 再訪意欲の高さ |
| `filter_service` | 配信サービスで絞り込み（`service`付き） | どのサービス目的の来訪が多いか |
| `change_season` | シーズン切替（`season`/`year`付き） | 過去クールの需要 |

### 自前の効果測定ダッシュボード（`/admin/analytics`・2026-07-13導入）

Vercel Proに課金せず、既に導入済みのSupabase（無料枠のPostgres）へ同じイベントを自前で記録し、
`/admin/analytics?token=...`で直近30日の集計を見られるようにした。追加費用ゼロ。

**未設定の間の挙動**: `analytics_events`テーブルが無い・`ADMIN_DASHBOARD_TOKEN`が未設定の間は、
既存の`watched`/`notify`機能と同じ設計思想で「静かにスキップ」する（計測が失敗してもユーザー操作
自体は壊れない）。ダッシュボードは`ADMIN_DASHBOARD_TOKEN`未設定なら「未設定です」と表示するだけ。

**有効化手順**:
1. Supabaseダッシュボードの **SQL Editor** で以下を実行（`docs/supabase-setup.md`の手順6と同じ場所）:
   ```sql
   create table if not exists public.analytics_events (
     id bigint generated always as identity primary key,
     event_name text not null,
     event_data jsonb,
     created_at timestamptz not null default now()
   );

   alter table public.analytics_events enable row level security;
   -- ポリシーは意図的に1つも作らない（anon/authenticatedからは読み書き不可。
   -- service_role（RLSを無視する）経由の /api/track・/admin/analytics からのみ操作する）。
   ```
2. `.env.local`（および本番Vercelの環境変数）に `ADMIN_DASHBOARD_TOKEN` を追加する
   （任意の推測されにくい文字列。パスワード相当なので、コミットや共有はしない）。
3. `/admin/analytics?token=<設定した値>` にアクセスして確認する。トークン不一致・未指定は
   404（存在しないページと同じ扱い。不一致の理由は教えない）。
4. `robots.txt`で`/admin`はクロール除外済み、ページ自体にも`noindex`を設定済み。

**記録している内容**: `event_name`（`share_site`等、コード側でホワイトリスト済みの8種のみ）と
`event_data`（`service`名等の付随情報）のみ。IPアドレス・Cookie・ユーザーIDは一切記録しない。

これらは「どの作品・どのサービスに関心が集まるか」を早期に掴むための土台。
広告を検討する際の入稿ターゲット（訴求する作品・サービス）の根拠に使う。
