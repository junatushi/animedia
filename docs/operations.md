# 運用手順書（runbook）

「アニメ視聴ガイド」を継続運用するための定期作業まとめ。
何を・いつ・どうやるかを固定し、担当が変わっても回るようにする。

シーズンの区切り（目安）:
- 冬 1月 / 春 4月 / 夏 7月 / 秋 10月 開始。各クールとも開始月の頭に切り替わる。

---

## サイクル早見表

| タイミング | やること | 使うもの |
|---|---|---|
| 新クール開始の直前〜当日 | ①データ点検 → ②新シーズン告知の投稿 | season-updater / sns-templates.md |

## ⑧ 作品詳細コンテンツの追記（あらすじ・見どころ・出版社・2026-07-07導入）

作品個別ページ（`/anime/[id]`）の「あらすじ」「見どころ」「出版社」はAnnictに構造化データが
無いため、公式サイト等の一次情報を人が確認しながら`content/works/{annictId}.json`に
1作品1ファイルで追記する運用にしている（声優・監督・製作会社・原作者名はAnnictから自動取得）。

**情報源の優先順位（一次情報のみ・Wikipediaは使わない）**:
- Wikipediaは一次情報ではないため使用しない（本ルールの`sourceUrl`＝一次情報という前提に反する）。
- **第1候補: 公式サイト**（Annictの`officialSiteUrl`）。あらすじ・見どころ・原作/出版社が明記されている。
- **第2候補: dアニメストアの作品ページ**。公式サイトがアクセス制限（403等）で取得できない場合の
  自動フォールバック。配信サービス自身が公開している紹介文なので一次情報として扱える。
  Claude単独で作品ページを特定できる（下記の手順で完全自動化可能）。

**dアニメストアから作品ページを特定する手順（公式サイトが取れない時）**:
1. 検索REST APIを叩く（ログイン不要・JSONが返る）:
   `https://animestore.docomo.ne.jp/animestore/rest/WS000105?searchKey=<作品名をURLエンコード>&length=20&vodTypeList=svod_tvod`
   → 応答に `workId` と作品タイトルの一覧が入る。タイトル（シーズン数含む）が一致するものを選ぶ。
2. 作品ページを取得する:
   `https://animestore.docomo.ne.jp/animestore/ci_pc?workId=<workId>`
   → あらすじ・原作・制作会社が公開されている。これを`sourceUrl`にする。

**進め方（セッション単位・完全自動化はしない＝内容の正確性は人の目でも確認する）**:
1. ユーザーが「続きをお願い」等と伝えたら、未整備の作品（`content/works/index.ts`の
   `WORK_DETAILS`に無いID）から目安15件を選ぶ。今期の注目度（`watchers`）が高い作品を優先する。
2. 上記の優先順位で一次情報を確認し、事実に基づいてあらすじ・見どころを要約する。
   **書いていない情報を創作しない**（CLAUDE.mdの「推測データで埋めない」方針と同じ）。
   著作権に配慮し、紹介文を丸写しせず要約する。出版社は一次情報に明記がある場合のみ記載し、
   アニメオリジナル作品や記載が無い場合は`publisher`を省略する。
3. `content/works/{annictId}.json` を作成し（`WorkDetailContent`型: synopsis / highlights /
   publisher? / sourceUrl）、`content/works/index.ts`の`WORK_DETAILS`に1行追記する。
   `sourceUrl`には実際に参照した一次情報のURL（公式サイトまたはdアニメの作品ページ）を必ず入れる。
4. 追記後は `npx tsc --noEmit` で型エラーが無いことを確認する。

### AI独断解釈サムネ（2026-07-07導入）

作品カード・作品ページに出すサムネは、権利者の画像を使わず、Pollinations（無料・APIキー不要）で
**タイトルから連想した本作品と無関係な創作イラスト**を事前生成して `public/works/{annictId}.jpg` に
保存している（表示時コスト・キー・レート制限ゼロ）。必ず「本作品との関連性はありません」の注釈を表示する。

**増やす手順**:
1. `scripts/gen-thumbnails.js` の `PROMPTS` に `{ id, seed, prompt }` を追記する。
   - プロンプトは**実在キャラの再現を避け**、タイトルの字面から連想した抽象的な情景にする。
   - **作品ごとに画風（絵柄トーン）を変える**（例: 水彩・油彩・浮世絵・フラットベクター・サイバーパンク等）。
2. `node scripts/gen-thumbnails.js` を実行（`public/works/` に保存し、`content/works/imageIds.ts` を自動更新）。
3. **生成画像を必ず人の目で確認**し、実在キャラに酷似していないか・不適切でないかをチェックしてからコミットする。
   気に入らなければ `seed` を変えて再生成する。

## ⑦ 週次ダイジェスト（フォロワー獲得施策・2026-07-05導入）

`.github/workflows/weekly-digest.yml` が毎週月曜21:00 JSTに自動実行し、
「今期の注目作TOP5」ダイジェストを生成する（`scripts/lib/build-digest.js`。
デプロイ済みサイトの `/api/season` を叩くだけで、ANNICT_TOKENの複製は不要）。

- **Bluesky・Mastodon**: 投稿APIが無料のため**自動投稿**する。Secrets未設定の間は
  「未設定のためスキップ」とログを出すだけで、ワークフロー自体は失敗にならない。
- **X**: 2026年2月の料金改定でAPI投稿が有料（1件$0.015〜$0.20）になったため
  自動投稿はしない。代わりに同じ下書き文を**GitHub Issueとして自動起票**するので、
  中身を確認してブラウザ/アプリから手動でコピペ投稿する。

### 有効化する場合に必要なSecrets（任意・未設定でも動く）
| サービス | Secret名 | 取得方法 |
|---|---|---|
| Bluesky | `BLUESKY_IDENTIFIER` | ハンドル名（例: `animedia.bsky.social`） |
| Bluesky | `BLUESKY_APP_PASSWORD` | Bluesky設定 → プライバシーとセキュリティ → アプリパスワード（**通常のログインパスワードではない**） |
| Mastodon | `MASTODON_INSTANCE_URL` | 登録したインスタンスURL（例: `https://mstdn.jp`） |
| Mastodon | `MASTODON_ACCESS_TOKEN` | インスタンスの 設定 → 開発 → 新規アプリ作成 → 「投稿」権限付きトークン |

いずれも**無料アカウント・無料の範囲**で取得できる（Xのような従量課金は発生しない）。

## ①③ データ点検（season-updater）

Claude Code で `season-updater` エージェントを起動し、対象シーズンを指定する。
確認できること:
- 対象シーズンの作品数（前シーズン比）
- 「配信情報なし」の作品一覧
- それが「Annict未登録」か「マッピング漏れ」かの切り分け

> 開始直後は配信情報が空の作品が多いのが**通常**（Annictの登録待ち）。
> 慌てて推測データで埋めない。2〜3週間後に再点検すれば大半埋まる。

## ⑤ マッピング点検（service-mapper）

`service-mapper` エージェントを起動。実データのチャンネル名を集計し、
`other`（その他配信）に落ちている中に、本来は配信サービス／放送局として
扱うべき名前がないか洗い出して `lib/services.ts` へ反映する。
反映後は必ず `node scripts/check.ts` が全件OKになることを確認する。

## ②④⑥ Xへの投稿（現在は手動投稿のみ・API自動投稿は保留中）

**2026-07-07判断: 自動投稿は再開せず、手動のまま投稿頻度を毎日に上げる運用にした。**
ネタ切れで続かなくなるのを防ぐため、[`docs/sns-templates.md`](./sns-templates.md)の
「5. 毎日投稿用ローテーション」に曜日ごとの投稿の型を用意している（迷ったら型に当てはめるだけでよい）。
新シーズン告知・バグ修正報告などのスポット投稿がある日はそちらを優先し、ローテーションは休む。

**2026-07-05判断: X APIでの自動投稿は保留し、ブラウザから手動で投稿する運用にしている。**
理由は、2026年2月にXが新規開発者向けの無料枠を廃止し完全従量課金制（投稿1件あたり
リンク無し$0.015・**リンクあり$0.20**）に切り替えたため。本ワークフローはURLを必ず
含める設計のため実質1投稿=$0.20。初回特典の$10クレジットで約50回分はあるが、
それ以降は追加課金が必須になる。ユーザーの判断で「まず手動投稿のみ」とした。

`.github/workflows/post-to-x.yml`（GitHub Actionsでの自動投稿）は**コードとして残してはいるが実行しない**。
GitHub Secrets（`X_API_KEY`等）も意図的に未登録のままにしておく。

### 実際の投稿手順（手動）
1. [`docs/sns-templates.md`](./sns-templates.md) から用途に合う文面をコピー
2. `{URL}` を実際のリンク（できればシーズンのディープリンク
   例: `https://animedia-khaki.vercel.app/?year=2026&season=summer`）に置換する
3. 数字（件数等）を入れる場合は投稿直前に実サイトで確認してから入れる
4. X（ブラウザ／アプリ）から自分で投稿する

### API自動投稿を再検討する場合
- $10クレジットの範囲で試す、またはリンクを外して$0.015/投稿に下げる等、コスト条件を
  再確認してからにする（本ドキュメント作成時点の料金。実施前に最新料金を再確認すること）。
- 再開する場合は GitHub Secrets（`X_API_KEY` / `X_API_SECRET` / `X_ACCESS_TOKEN` / `X_ACCESS_SECRET`）
  の登録が必要。未登録だとワークフローはエラーで止まる（＝誤って課金される事故は起きない設計）。

## 計測の見かた（マーケ）

**2026-07-05判断: Vercel Web Analyticsは無料（Hobby）プランで有効化している。**
無料プランはページビュー・訪問者数・直帰率等の基本統計のみで、**カスタムイベントは記録されない**
（Vercel側の仕様。Proプラン＝月$20が必要）。下記のカスタムイベント（`share_site`等）を送る
`track()`呼び出しは`app/page.tsx`にコードとして残しているが、無料プランの間はダッシュボードに
表示されない（エラーにはならず無害）。Proへの課金判断があれば、そのまま有効になる。

Vercel の Analytics ダッシュボードで、ページビューに加えて以下のカスタムイベントを確認できる（要Proプラン）:

| イベント | 意味 | 見るポイント |
|---|---|---|
| `share_site` | サイト全体をXで共有 | 拡散のきっかけ数 |
| `share_work` | 作品単位でXで共有（`title`付き） | どの作品が共有されるか＝話題性 |
| `favorite_add` | お気に入り登録 | 再訪意欲の高さ |
| `filter_service` | 配信サービスで絞り込み（`service`付き） | どのサービス目的の来訪が多いか |
| `change_season` | シーズン切替（`season`/`year`付き） | 過去クールの需要 |

これらは「どの作品・どのサービスに関心が集まるか」を早期に掴むための土台。
広告を検討する際の入稿ターゲット（訴求する作品・サービス）の根拠に使う。

## 依存パッケージの方針

### Next.js は 14.2.35 で維持中（2026-07-05判断、保留中）
`npm audit` の critical は 14.2.5→14.2.35 のパッチ更新で解消済み。残る high/moderate は
next/image最適化・middleware・Server Actions・Pages Router i18n に関するものだが、
本アプリはこれらをいずれも使っていないため実質非該当と判断し、Next 16へのメジャー
アップグレード（`params`/`searchParams`非同期化・fetchキャッシュ挙動変更・React 19必須化等、
稼働中サイトを壊しうる破壊的変更が多い）は見送っている。

**再検討するタイミング**（いずれか発生したら再提案する）:
- `npm audit` で新たな脆弱性が出た、または上記機能（next/image等）を実際に使い始める
- Next.js 14系のセキュリティサポートが終了する
- ユーザーが再検討したいと言った場合
